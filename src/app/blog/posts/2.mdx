# 단골손님 - 소상공인 구독 서비스 플랫폼

## 프로젝트 목적

대기업 프랜차이즈에서는 이미 정기 구독 서비스를 통해 단골 고객을 유치하고 있습니다. 하지만 동네의 작은 식당이나 카페는 이러한 시스템을 도입하기 어려운 경우가 많습니다. **단골손님은 소상공인을 위한 맞춤형 구독 서비스 중개 플랫폼으로, 손님에게는 저렴한 가격으로 지속적인 서비스를 받을 기회를, 소상공인에게는 안정적인 매출을 제공하는 것이 목표입니다.**

## 핵심 목표

- **소상공인용 구독권 관리 기능**
- **손님용 정기/횟수 차감 구독권 결제 기능**
- **QR 코드를 통한 구독권 조회 기능**

## 프로젝트 개요

- **프로젝트명**: 단골손님
- **설명**: 소상공인을 위한 구독 서비스 중개 플랫폼
- **배포 링크**: 운영 중단
- **깃허브**: [프로젝트 GitHub 링크](https://github.com/jaemisam-a/dangol-sonnim-backend)
- **작업 인원**: 백엔드 1명(본인) / 프론트엔드 2명 / 디자이너 2명

## 주요 기능

### 1. 사장님 전용 가게 및 구독권 관리 기능
- 가게를 등록하고 직접 구독권을 생성 및 관리할 수 있음
- 구독권별 사용 가능 횟수 및 유효 기간 설정 가능
- 구독권 판매 현황을 실시간으로 확인할 수 있음

### 2. 손님 전용 구독권 결제 기능
- 정기 구독과 횟수 차감형 구독권 두 가지 유형 제공
- 신용카드 및 카카오페이와 같은 결제 수단 지원
- 마이페이지에서 현재 활성화된 구독권 확인 가능

### 3. QR 코드로 구독권 조회 기능
- 오프라인 매장에서 QR 코드 스캔을 통해 구독권 인증 가능
- 사장님이 손님의 구독권을 실시간으로 확인 가능하여 간편한 결제 처리 가능

## 담당 업무

- **ERD 설계**: 기획 요구사항을 바탕으로 주도적으로 데이터베이스 설계 진행
- **이메일 인증 기반 비밀번호 변경 기능 구현**
- **백엔드 개발 기여도 90%**
- **HateOAS 적용을 통한 API 사용성 향상**
- **RestDocs를 활용한 API 문서화 및 테스트**

## 성과 및 배운 점

### 고급 연관 매핑 적용으로 중복 코드 개선
- **JPA 상속 매핑 (`@Inheritance(strategy = InheritanceType.SINGLE_TABLE)`)을 활용하여 공통 엔티티 속성 통합**
- **중복된 CRUD 로직을 공통 서비스 계층으로 분리하여 유지보수성 강화**

### RESTful API 향상 (HateOAS 적용)
- **HateOAS 적용 전**: 단순 JSON 응답 구조로 API 확장성이 부족함
- **HateOAS 적용 후**:
- 각 리소스에 대한 추가적인 링크 정보를 포함하여 API 사용성을 향상
- 클라이언트가 API 문서를 참고하지 않아도 직관적으로 사용 가능

### RestDocs를 활용한 API 문서화 및 테스트
- **API 문서화 자동화**: RestDocs를 사용하여 실제 API 테스트를 기반으로 문서 자동 생성
- **개발 생산성 향상**: API 변경 시 자동으로 문서 업데이트되어 유지보수 용이

### **팀원 이탈 후 혼자서 끝까지 프로젝트를 완수한 경험**
- 초기 백엔드 팀원이 3명이었지만, 중간에 취업 등의 이유로 팀원이 빠지면서 혼자 백엔드를 담당하게 됨
- 디자이너와 프론트엔드 팀원들이 남아 있어 프로젝트를 포기할 수 없었고, 끝까지 책임감을 갖고 개발을 완료함
- 어려운 상황에서도 책임감으로 프로젝트를 완수하고 배포한 경험을 통해 성장함
